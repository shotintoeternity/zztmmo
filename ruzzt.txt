Thought for 2m 32s

Recommendation (short): run a server-authoritative RUZZT engine (headless) for the simulation and a WASM browser client for rendering + input. Start with a one-active-player MVP (others spectate), then extend RUZZT’s Player logic to support N players (each consumes a stat slot) and replicate tile/stat diffs each tick via WebSockets. Treat each board (60×25) as a “room/shard”, keep RNG/timing deterministic, and persist using RUZZT’s file/serde support.

How to make ZZT multiplayer with RUZZT
1) What RUZZT already gives you
Separation of concerns: ruzzt_engine is a reusable simulation library; ruzzt is the SDL2 desktop front end. There’s also zzt_file_format (serde-friendly) and a zzt_web_editor that already compiles to WASM and renders worlds in the browser—perfect as a starting client.
World/board I/O: RUZZT can load/write ZZT/SZT via zzt_file_format (RLE board data, etc.).
2) Architecture to aim for

Authoritative server (Rust, tokio + WebSockets):

Run one headless ruzzt_engine instance per active board (“room”). ZZT boards are 60×25; status elements (“stats”) are the active entities list. Default ZZT limit is ~150 stats per board (player usually at index 0).
Tick at a fixed rate (≈9 Hz like classic ZZT; choose a clean integer like 10 Hz if you prefer—but keep it consistent).
Capture inputs from clients → apply in a deterministic order before each tick → advance world → emit diffs (changed tiles + stat field changes + HUD).

Browser client (WASM):

Reuse the zzt_web_editor WASM build path to embed ruzzt_engine for rendering/decoding tiles/fonts and for local interpolation if desired. Hook a WebSocket for input and state updates.
For audio, synthesize simple PC-speaker beeps in WebAudio (there is no Web “PC speaker” API).

Why not lockstep? It’s viable for tiny co-op, but an authoritative server gives you better scaling, cheat resistance, and easier late joins.

3) Network model (practical)

Messages (protobuf/JSON/borrowing serde):

Client → Server: Join {name}, Input {seq, held: KeyMask}, Chat, ReadyForBoard {id}
Server → Client: Snapshot {board_id, seed, stat_count, tiles[..], stats[..]}, Diff {frame, tile_ops[..], stat_ops[..], hud}, Chat, BoardTransition {to}

Diffs:

Tiles are tiny; send RLE spans (y, x_start, len, elem, color) or sparse (x,y,elem,color) ops. (It mirrors ZZT’s own RLE approach and keeps payloads tiny.)
Stats: send only fields that changed (position, cycle, ip, params, etc.).

Joining mid-game: send a full Snapshot then diffs.

4) MVP (two sprints)

Sprint A — “Twitch Plays ZZT”

One active player; everyone else spectates. Server runs ruzzt_engine, steps at fixed Hz, streams diffs. RUZZT’s README notes speed control is missing and there are sleeps—strip/sidestep those on the server and drive ticks with Interval.

Sprint B — hotseat co-op

Add a turn token or “request control” queue so multiple clients can play without engine changes. Minimal risk, validates net code and state replication.
5) True co-op (multiple players on a board)

To get simultaneous players you’ll need a small, surgical extension to ruzzt_engine:

Multiple players: ZZT assumes one Player stat. Implement PlayerN (N≥1) or let the engine maintain a Vec<PlayerRef> so input can be routed per player. Each consumes a stat slot, so budget against the ~150-stat cap.
Collisions/interaction: share “player movement & interaction” code path across players; ammo/keys/torches are per-player HUD values; gems/scrolls affect local player unless world scripts say otherwise.
Board transitions: elect a leader policy (e.g., transition when any player touches an edge, and teleport others to the board’s entry if reachable) or require majority near edge for cooperative transitions.
ZZT-OOP semantics: messages like #GIVE/#ZAP that target “the player” need rules (nearest player on board, or always leader). Start with “nearest on same board” and document it.

Tip: the stat list iteration order matters in classic ZZT; keep your multi-player stats ordered stably to avoid Heisenbugs in object messaging.

6) Determinism & RNG
Seed the engine per board instance on server; include {seed, frame} in snapshots so resims stay bit-exact.
Run a single RNG stream inside the server sim; clients never RNG. (If you ever try input-lockstep, you must guarantee same seed and tick order for all peers.)
7) Persistence & scale
Sharding: one process per world; one engine instance per active board; idle boards can unload and serialize state through zzt_file_format (serde) to your DB/object storage.
Limits: world has ≤101 boards (title + 100). For MMO scale, stitch multiple world files into a larger “overworld” registry and instance boards for large groups.

Server (sketch):

// server/src/world.rs
struct Room {
    world: ruzzt_engine::World,
    last_frame: u64,
    last_sent: CompressedState,
}
impl Room {
    fn tick(&mut self, inputs: &[Input]) -> Diff {
        apply_inputs(&mut self.world, inputs);
        self.world.step();            // advance one ZZT tick
        let diff = make_diff(&self.world, &self.last_sent);
        self.last_sent = compress(&self.world);
        diff
    }
}

Client (sketch):

// client_wasm/src/lib.rs
#[wasm_bindgen(start)]
pub async fn start() {
    let ws = connect_ws().await?;
    let mut renderer = WasmRenderer::new(); // reuse zzt_web_editor render pieces
    loop {
        match ws.recv().await? {
            ServerMsg::Snapshot(s) => renderer.load_snapshot(s),
            ServerMsg::Diff(d)     => renderer.apply_diff(d),
        }
    }
}

(You can lift fonts/renderer bits from zzt_web_editor and keep them in sync with engine serialization.)

9) Checklist & gotchas
Fix tick rate (remove sleeps; drive via tokio::time::interval). RUZZT notes speed control is missing; that’s fine server-side.
Input canonicalization: map browser keys to ZZT directions/shoot; coalesce repeats into a per-tick keymask.
Audio: synth in WebAudio (square wave) since “PC speaker” isn’t a web thing.
Bandwidth: A full board is 1500 tiles; diffs per tick are typically tiny. RLE and stat-field deltas keep packets small.
Compatibility: standard worlds expect one player; advertise “co-op mode may alter vanilla semantics” and curate a co-op world set.
