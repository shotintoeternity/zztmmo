Thought for 6s

Verdict: Zeta will make it easier to get ZZT running in a browser quickly, but it won’t make it easier to build a true MMO version of ZZT. It’s a great client/runtime foundation, not an MMO framework.

Why Zeta helps (a lot)

Zeta is explicitly a small emulator that implements just enough DOS-ish environment to run ZZT/Super ZZT accurately in modern environments, including WebAssembly/HTML5 embedding. That’s huge if your goal is “play ZZT worlds on the web fast.”

The zeta-z2 HTML5 build is basically “drop in a canvas + load a zip” with ZetaLoad(...), plus optional persistent storage settings—so you can ship a web client quickly.

Why Zeta doesn’t make MMO easier

An emulator runs the original single-player game loop. An MMO wants things like:

authoritative server state
replication/delta updates
anti-cheat / validation
synchronization rules (latency hiding, rollback, lockstep, etc.)
a multiplayer-aware simulation model

With Zeta, you’re fighting the shape of the problem: you’d be “bolting networking onto a DOS ZZT process.” That usually means ugly hooks into memory/state, custom I/O virtualization, or running multiple emulated instances server-side and trying to reconcile them. It’s doable, but it’s more work than it sounds, and tends to get brittle.

The practical sweet spot

If your MMO vision is closer to “ZZT as a shared online experience” (lobbies, world sharing, spectators, chat, asynchronous interactions, maybe per-room instancing), Zeta is awesome: you can treat it like a drop-in ZZT runtime and build the online layer around it.

If your MMO vision is “many players in the same board at once affecting the same simulation”, Zeta is more like a prototype client while you build the real thing as a multiplayer-native engine (reimplementation / port), because you’ll eventually want clean control over state, determinism, and netcode.

A sane strategy (if you want velocity)
Use Zeta now for: instant browser client, compatibility, content pipeline, quick playable demos.
Design MMO server + protocol separately, and decide early whether:
players are watching/instanced (Zeta-friendly), or
players are co-simulating (engine-rewrite-friendly).

Why Zeta shouldn’t be your multiplayer core

Zeta runs a single-player DOS process. For “multiple players on the same board,” you need:

one authoritative board state
deterministic tick progression
validated player actions
replication/deltas to many clients
consistent conflict resolution

Trying to retrofit that into an emulator usually turns into brittle memory hooking + desync hell.

How we do incorporate Zeta into the plan
1) Golden reference + regression testing (huge)

Use Zeta/Reconstruction as the ground truth for:

ZZT-OOP semantics (timing, object order, edge cases)
board transitions, message behavior, random number quirks
“does this world behave like classic ZZT?”

Practical pattern: build an automated test harness that runs a world in Zeta for N ticks, snapshots key state (board tiles, object list, stats, RNG state if possible), and compares against your server engine’s snapshots. This is how you avoid “we rewrote it and it feels off.”

2) Content pipeline & compatibility target

Keep shipping/validating .ZZT worlds against Zeta early:

Your editor/import/export pipeline can be validated by “does it run in Zeta?”
Any extension features you add can be gated behind flags so classic worlds remain intact.
3) “Reference client” for early UX + rendering

Even if Zeta isn’t the authoritative sim, it’s still useful for:

quickly rendering boards
verifying palette/charset/tile rendering fidelity

(But for multiplayer, you likely won’t run the sim inside Zeta on clients.)

What the multiplayer architecture should look like
Authoritative server: a ZZT runtime you control

Implement a ZZT VM/runtime on the server that owns:

board state (tiles, colors, passages, scrolls)
object list + execution order
tick loop
RNG (seeded, deterministic)
player entities (multiple “player objects” with ids)

Key design call: decide whether ZZT “player” is just another object type (recommended) with input-driven movement, collisions, shooting, etc.

Clients: dumb(ish) terminals

Clients do:

render the board + objects
send input events (movement/fire/chat)
receive state updates/deltas

This makes cheating harder and desync manageable.

Networking model: server-tick + input queue
Fixed tick rate (classic-ish), e.g. 10–20 ticks/sec (you can tune)
Clients send inputs tagged with client time; server applies on next tick
Server broadcasts either:
full state (easy, heavier), or
delta updates (harder, scalable)

Start with full snapshots at low tick rate, graduate to deltas once it works.

The “ZZT multiplayer problem” you need to resolve

Classic ZZT assumes one player. With many:

collisions: what if two players step into the same tile same tick?
bullets: ownership, hits, friendly fire?
interaction priority: who activates an object/scroll?
message/scroll UI: per-player, not global (critical)

Rule of thumb: make anything UI-ish per-player, and anything physical board-ish server-global.

Why wouldn’t I just use ZZT for that?

Because classic ZZT is a single-player DOS program. You can “use ZZT” as the simulation, but the moment you want many players in one board, you’re forced into one of two ugly paths:

1) “Run one ZZT and hack it into multiplayer” (hard + brittle)

You’d have to retrofit things ZZT never had:

Multiple player entities: ZZT has one “you.” You’d need extra player objects, input routing, per-player health/ammo/gems, etc.
Per-player UI/scrolls: ZZT’s message system (scroll window, text prompts, “press any key”) is global. In multiplayer, one player reading a scroll can’t freeze everyone.
Authoritative state + anti-cheat: If clients are running their own ZZT, you can’t trust them. If a server runs ZZT, you still need to validate actions and prevent “teleport/memory-edit.”
Networking + determinism: You’d need to make the sim deterministic under multiple inputs and network latency.

Doing that inside a DOS binary means patching memory, intercepting keyboard input, rewriting rendering/UI paths, and hooking object logic. It’s doable in principle, but it’s the kind of “weeks become months” engineering where every edge case is pain.

2) “Run one ZZT per player and sync them” (desync city)

This is the lockstep/replication approach: everybody runs ZZT locally, and you try to keep them identical by sharing inputs.

Problem: ZZT was not built for this. Any mismatch in:

random number usage
timing/tick scheduling
object execution order
input arrival jitter

…creates divergence. Once you diverge, you’re basically dead unless you implement rollback + state sync—and ZZT doesn’t expose clean state serialization for that.

The deeper reason: ZZT’s UX model conflicts with MMO

Even if you could sync the board, ZZT’s interaction model assumes “the whole game is one screen controlled by one person.”

scrolls / prompts / “game paused” moments
“one cursor focus”
“press any key” gates
assumptions about who owns “the player” stats

For multiplayer, you need to split “world state” from “player state” cleanly. Classic ZZT mixes them.

So what’s Zeta/Reconstruction good for again?
As a golden reference: “our reimplementation matches real ZZT behavior.”
As a classic single-player mode you can ship alongside MMO mode.
As an execution harness for compatibility testing and world validation.
If you still want to “just use ZZT,” the least-bad way

Run one server-side instance as the authority and stream it—basically like remote desktop with multiple controllers. But then:

only one person can really “be the player” at a time
others are spectators or take turns
it’s not truly “multiple players inside the same board”

So: you can use ZZT, but not without turning it into something it wasn’t designed to be. If your goal is genuine co-presence, a controlled server simulation (rewrite) is the straight path; ZZT/Zeta are your correctness oracle and compatibility layer.
